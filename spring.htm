Repo-sizing grep commands (Instancio fixture-viability check)

1. Total types (denominator):
grep -rEl "(class|interface|enum|record)[[:space:]]+[A-Za-z_]" --include=*.java . | wc -l

2. Interfaces / abstract classes:
grep -rEl "interface[[:space:]]+[A-Za-z_]" --include=*.java . | wc -l
grep -rEl "abstract[[:space:]]+class[[:space:]]+[A-Za-z_]" --include=*.java . | wc -l

3. Distinct interface/abstract type names (the real cost number):
grep -rhoE "interface[[:space:]]+[A-Za-z_][A-Za-z0-9_]*" --include=*.java . | awk '{print $NF}' > iface_types.txt
grep -rhoE "abstract[[:space:]]+class[[:space:]]+[A-Za-z_][A-Za-z0-9_]*" --include=*.java . | awk '{print $NF}' >> iface_types.txt
sort -u iface_types.txt -o iface_types.txt
wc -l iface_types.txt

4. Exposure (classes with a field of one of those types):
while read T; do
grep -rlE "private[[:space:]]+(final[[:space:]]+)?$T[[:space:]]+[A-Za-z_]" --include=*.java .
done < iface_types.txt | sort -u | wc -l

5. Generic class declarations:
grep -rEl "class[[:space:]]+[A-Za-z_][A-Za-z0-9_]*[[:space:]]*<" --include=*.java . | wc -l

 

 

 

 

 

 

 

 

 

 

 

 

 

DUO test-coverage kit: gap extractors + prompts (added 2026-07-15)

0. Context / when to use

Goal: drive a Java class's UNIT + MUTATION coverage up to completion with an LLM (DUO in the IDE).
Two scenarios:
  A. The test class was created by the testgenlite v2 flow (JavaPoet skeleton + Instancio/Mockito).
  B. A test class already existed (no JavaPoet/Instancio) - leverage it and add coverage.
Point of truth = IntelliJ: Run with Coverage (JaCoCo gutter) + PITest plugin/Gradle (surviving mutants).
KEY WARNING: the "fix the compile/test errors" retry prompt DELETES tests it cannot make pass
(coverage goes DOWN, e.g. 21 tests -> 8). To RAISE coverage you must switch to a GAP-DRIVEN prompt
(below) that only APPENDS tests, driven by real JaCoCo/PITest gaps.

1a. Extract PITest survivors (from build/reports/pitest/mutations.xml)

import re
xml = open('build/reports/pitest/mutations.xml', encoding='utf-8').read()
for m in re.findall(r'<mutation\b.*?</mutation>', xml, re.S):
    st = re.search(r"status='(\w+)'", m).group(1)
    if st in ('SURVIVED', 'NO_COVERAGE'):
        line = re.search(r'<lineNumber>(\d+)</lineNumber>', m).group(1)
        meth = re.search(r'<mutatedMethod>([^<]+)</mutatedMethod>', m).group(1)
        mut  = re.search(r'\.(\w+)</mutator>', m).group(1)
        desc = re.search(r'<description>([^<]+)</description>', m).group(1)
        print(f"{st}  {mut}  line {line}  {meth}: {desc}")

1a. Example output (real, from a SequenceGroups run)

SURVIVED  VoidMethodCallMutator  line 46  addSequences: removed call to Sequence::set
SURVIVED  VoidMethodCallMutator  line 55  addSequences: removed call to Sequence::set

1b. Extract JaCoCo uncovered lines/branches (from build/reports/jacoco/test/jacocoTestReport.xml)

import xml.etree.ElementTree as ET
t = ET.parse('build/reports/jacoco/test/jacocoTestReport.xml')
for sf in t.iter('sourcefile'):
    if sf.get('name') == 'Util.java':          # <-- your SUT's .java file name
        for ln in sf.findall('line'):
            mi = int(ln.get('mi', 0)); mb = int(ln.get('mb', 0))   # mi=missed instr, mb=missed branch
            if mi > 0 or mb > 0:
                print(f"UNCOVERED  Util line {ln.get('nr')}  missedInstr={mi} missedBranch={mb}")

1b. Example output (real)

UNCOVERED  Util line 24  missedInstr=3 missedBranch=0

(In IntelliJ without scripting: the red/yellow coverage gutter is the same JaCoCo data; the PITest
plugin's "survived" list is the same mutations.xml. The scripts just make it copy-pasteable.)

2. The GAP-LIST FORMAT that works (survivor + one-line KILL HINT) -- this drove 30% -> 100% branch on LiteTimeoutBlockingWaitStrategy

SURVIVED  VoidMethodCallMutator  line 46  removed call to SequenceBarrier::checkAlert
   -- Kill: AlwaysAlertedBarrier + cursor behind target so the synchronized loop is entered;
      assertThrows(AlertException.class, () -> waitStrategy.waitFor(...)).
SURVIVED  PrimitiveReturnsMutator  line 61  replaced returned long with 0
   -- Kill: cursor & dependentSequence already = 3, sequence = 3; assert waitFor returns EXACTLY 3.
NO_COVERAGE  line 79  toString() never called
   -- Kill: assert toString() contains "LiteTimeoutBlockingWaitStrategy".

3. PROMPT - Scenario A (test class from testgenlite v2 / JavaPoet+Instancio; use AFTER it is green; STOP using the retry prompt)

You previously generated a JUnit test class. A real PITest + JaCoCo run found surviving mutants and
uncovered branches. Strengthen the test class to close them WITHOUT weakening anything working.

RULES
1. Keep every existing @Test that passes; only ADD new @Test methods (or @ParameterizedTest) to close the gaps.
2. Reuse the Instancio/Mockito fixtures already in the class. Use only types/methods visible in the SUT. Java 8 syntax. Never leave an ambiguous import (fully-qualify JDK-colliding names like TimeoutException).
3. For each gap use exact-value assertions - assertEquals(expected, actual), boundary and boundary+/-1, verify(...) for void calls. No assertNotNull / assertTrue(true).
4. One behaviour per test; name methodName_condition_expectedOutcome.
Output ONLY the complete revised test class. No fences, no prose.

CLASS UNDER TEST:
{{SUT_SOURCE  (with the dependency-graph comments if present)}}

CURRENT TEST CLASS:
{{CURRENT_TEST}}

SURVIVING / UNCOVERED (each line = one target, with a kill hint):
{{GAP_LIST_WITH_HINTS}}

4. PROMPT - Scenario B (a test class that already existed, no JavaPoet/Instancio; goal5 add-to-existing)

You are adding tests to an existing JUnit {4|5} test class to raise coverage. Preserve the file's
structure, imports and helpers. Add only NEW @Test methods that close the gaps listed. Do NOT modify
or delete existing tests. Reuse fixtures/helpers already present. Use only types/methods visible.
Java 8 syntax; no ambiguous imports.

For each gap:
- UNCOVERED BRANCH  -> a test taking the missing side of that condition.
- SURVIVED negated/removed conditional -> assert the two branches give DIFFERENT observable outcomes.
- SURVIVED replaced-return -> assert the EXACT expected value.
- SURVIVED removed void-call -> verify(...) the interaction or assert the state change.
- boundary mutant -> the exact boundary and boundary+/-1.

Output ONLY the new @Test methods, a NEW IMPORTS: block if needed, then a GAP-MAP: pairing each test
to its gap id.

### Class under test
{{CUT_SOURCE}}
### Existing test class (match its style exactly)
{{EXISTING_TEST}}
### Gaps to close (from JaCoCo + PITest)
{{GAP_LIST_WITH_HINTS}}

5. Step-by-step loop (both scenarios)

1. IntelliJ: Run with Coverage (JaCoCo) + run PITest on the class -> get uncovered lines + survivors.
2. Build the gap list, one line per item, and add a ONE-LINE kill hint per item (see section 2).
   Use the scripts in 1a/1b, or read the gutter + PIT panel.
3. Paste  SUT + current test + gap list  into DUO with the matching prompt (section 3 or 4).
   Append the returned @Test methods into the existing class (do not replace the whole file).
4. Re-run coverage + PITest. Repeat 1-3 on the remaining survivors.
5. STOP when the only survivors are EQUIVALENT MUTANTS - timing/liveness (awaitNanos, LiteTimeout),
   concurrency redundancy (e.g. a value set twice for thread-safety). Those are documented dead-ends,
   not failures; chasing them yields flaky tests.

6. Errors / gotchas (and fixes)

- Coverage went DOWN after a round -> you used the "fix errors" retry prompt (it deletes failing
  tests). Switch to the gap-driven prompt above; it only appends.
- Ambiguous import in IntelliJ (Alt+Enter offers multiple candidates, e.g. TimeoutException) -> tell
  the LLM to fully-qualify JDK-colliding names; pick the CUT's own package when prompted.
- assertNotNull / assertTrue(true) leave mutants alive -> demand exact-value asserts (assertEquals,
  boundary+/-1, verify). Exact values are what convert line coverage into mutation kills.
- A generated test flakes under full-suite load (timing) -> widen the timing margin in that test;
  do not band-aid by deleting it.
- Self-reported GAP-MAP from the LLM can be wrong -> always re-run JaCoCo/PITest to verify a gap is
  actually closed; trust the tool, not the model's claim.
- Testability predicts effort: high-testability classes hit 100% first pass; low-testability /
  very-high-complexity classes plateau at ~73-88% (equivalent mutants) after 2+ gap-driven passes.
