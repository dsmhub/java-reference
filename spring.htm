Repo-sizing grep commands (Instancio fixture-viability check)

1. Total types (denominator):
grep -rEl "(class|interface|enum|record)[[:space:]]+[A-Za-z_]" --include=*.java . | wc -l

2. Interfaces / abstract classes:
grep -rEl "interface[[:space:]]+[A-Za-z_]" --include=*.java . | wc -l
grep -rEl "abstract[[:space:]]+class[[:space:]]+[A-Za-z_]" --include=*.java . | wc -l

3. Distinct interface/abstract type names (the real cost number):
grep -rhoE "interface[[:space:]]+[A-Za-z_][A-Za-z0-9_]*" --include=*.java . | awk '{print $NF}' > iface_types.txt
grep -rhoE "abstract[[:space:]]+class[[:space:]]+[A-Za-z_][A-Za-z0-9_]*" --include=*.java . | awk '{print $NF}' >> iface_types.txt
sort -u iface_types.txt -o iface_types.txt
wc -l iface_types.txt

4. Exposure (classes with a field of one of those types):
while read T; do
grep -rlE "private[[:space:]]+(final[[:space:]]+)?$T[[:space:]]+[A-Za-z_]" --include=*.java .
done < iface_types.txt | sort -u | wc -l

5. Generic class declarations:
grep -rEl "class[[:space:]]+[A-Za-z_][A-Za-z0-9_]*[[:space:]]*<" --include=*.java . | wc -l

 

 

 

 

 

 

 

 

 

 

 

 

 

 

Analysing a repo — comparing branches & deciding what to bring across

Pure git commands run in any shell (cmd, PowerShell, Git Bash). Commands piped through awk / sort / uniq are marked (Git Bash only) — they will not run in cmd or PowerShell; use the Git Bash shell bundled with Git for Windows. Replace main / feature with your two branch names; A..B means changes on B since the merge base, A...B is the symmetric difference.

What one branch has that another does not

Counts of commits each branch has that the other lacks — left = main-only, right = feature-only; the size of the gap at a glance:

git rev-list --left-right --count main...feature

Feature's commits marked + (not yet in main) or - (already applied, matched by patch-id) — your cherry-pick shortlist, and it catches ones already brought across under a different hash:

git cherry -v main feature

Readable summary of every commit in feature-but-not-main, grouped by author — a quick overview of what the branch actually contains:

git shortlog --no-merges main..feature

Quantify the difference

Every file that differs between the two branch tips, with per-file +/- bars and a grand total — the overall size of the change:

git diff --stat main..feature

Each differing file tagged Added / Modified / Deleted — what changed and how, without the diff body:

git diff --name-status main..feature

Files that exist on feature but not on main (added) — the net-new files unique to the branch:

git diff --name-only --diff-filter=A main..feature

Rank files — which carry the weight, which are contested

Files ranked by total lines changed across the range — the files carrying the most churn (decide these deliberately): (Git Bash only)

git log --numstat --format= main..feature | awk 'NF==3{a[$3]+=$1;d[$3]+=$2} END{for(f in a) printf "%6d  +%-6d -%-6d %s\n", a[f]+d[f], a[f], d[f], f}' | sort -rn | head -20

Files ranked by how many commits in the range touched them — the frequently-touched, often-contested files: (Git Bash only)

git log --format= --name-only main..feature | sort | uniq -c | sort -rn | head -20

Spot messy commits, and size the range

Each commit followed by its files-changed / insertions / deletions line — scan for the oversized commits that likely mix unrelated concerns:

git log --format='%h %s' --shortstat main..feature

Total lines added and removed across the whole range — one number for how big the bring-across is: (Git Bash only)

git log --numstat --format= main..feature | awk '{a+=$1;d+=$2} END{print "added="a"  removed="d}'

Target one file, or reconcile diverged series

Only the commits on feature (not main) that touched a given path — find exactly the commits to cherry-pick for that file:

git log --oneline main..feature -- path/to/file

Compare two similar commit series (e.g. before/after a rebase or a partial cherry-pick) to see what genuinely differs between them — for reconciling branches that have diverged:

git range-diff main feature~5 feature

 

Bringing a long-running feature branch into a restructured main (JDK 8→21 uplift)

A branch cut from main and worked on for weeks (a JDK 8→21 uplift, new tests, bug fixes, Sonar quality fixes). Meanwhile main was restructured from root test/<packages> + source/<packages> to source/main/java + source/test/java. Merging the old branch directly makes Sonar treat every file as new code — including files that only moved. Instead, cut a fresh branch (e.g. JDK21) from the restructured main and bring changes across by category, not by replaying history.

1. The four categories, and how each is handled

Clean bug-fix commits (3–5): cherry-pick by hash — preserved as discrete history. In IntelliJ, Git Log → filter to the source branch → multi-select → Cherry-Pick is nicer than the terminal here.

Config (build.gradle, libraries.gradle, CI YAML): take the final state, no replay.

WARNING — build.gradle must be HAND-MERGED, not taken wholesale. The old branch's build.gradle describes the old source-set layout; taking it wholesale overwrites the correct one and breaks the build. Take the toolchain / dependency changes; keep the new source sets.

Sonar quality fixes: write the branch version straight to the correct new location — this records as a modification to the existing file rather than an add-then-move, which matters for how Sonar classifies it:

git show <branch>:<old-path> > <new-path>

Test coverage (the bulk, 50–70+ files across packages): grab at the branch tip, move, single commit (section 4).

2. Key insight

Checking out from a branch name gives the file as of the branch tip — the final cumulative version with every earlier change already baked in. That sidesteps a file having been touched across many commits. Do NOT select from individual commits in the log.

3. Execution order

Config first (the build must be green before anything else is verifiable) → clean bug-fix commits → Sonar fixes → tests → verify → push once at the end.

4. The test-file move — check for package collision first

Both move commands below need Git Bash, not PowerShell/cmd. First list the branch's old test tree and the new one:

git ls-tree -r --name-only <branch> test/ | head -30
ls src/test/java

If NO overlap — move wholesale; git mv fails loudly on a collision, which is what you want:

git checkout <branch> -- test/
git mv test/* src/test/java/
git commit -m "move test coverage to new source set"

If OVERLAP — merge into existing packages, overwriting file-by-file (silent); only run when you are certain the branch version is wanted everywhere:

git archive <branch> test | tar -x --strip-components=1 -C src/test/java

Critical: do NOT commit between the grab and the move. Git then records only the final state as one clean commit, instead of an add followed by a rename.

5. Why not IntelliJ for the move

Dragging packages in the Project view triggers Move refactoring, which may rewrite package declarations and imports. The package names are not changing — only the physical folder — so refactoring is the wrong tool and messy to unpick.

6. Safety

Commit after each category so every step has a rollback point; anything that goes wrong is then a clean reset. Run the checkout and inspect git status before moving anything.

git status
git reset --hard HEAD

7. Verification before pushing

Build green, confirm nothing is left at the old paths, then review the whole change as a summary:

./gradlew clean build
git status
git diff main...JDK21 --stat

8. Afterwards

Keep and tag the old branch — do not delete it until confident. Merge JDK21 to main as a normal PR so Sonar runs on it before it lands.

 

Line endings (CRLF vs LF) — checking and managing them

Working on a Mac against a repo with Windows contributors, the risk is a line-ending flip that makes every line of a file read as changed. That is the same trap as the folder move in the section above (a change arriving “through a different door”): it would defeat the Sonar work, so line endings must be under control before and during any file move.

1. What is in the repo now

Count each line-ending type across tracked files. All i/lf is healthy; a mix of i/lf and i/crlf means it is already inconsistent. (Git Bash)

git ls-files --eol | awk '{print $1}' | sort | uniq -c

2. What the local machine does to files

Shows the local checkout/commit conversion setting:

git config --get core.autocrlf

On macOS: you want input (commit LF, no checkout rewriting) or unset. Never true — that is the Windows setting and it rewrites files on checkout. On Windows the conventional value is true, but with the .gitattributes in step 4 the repository enforces LF regardless of anyone's local setting, which is what actually keeps it consistent.

3. Whether the repo defends itself

If this file is absent, each developer's local config decides behaviour — that is how repos drift:

cat .gitattributes

4. The fix — a .gitattributes at the repo root

Forces LF in the repository regardless of who commits from what:

* text=auto eol=lf
*.bat text eol=crlf
*.cmd text eol=crlf
*.sh text eol=lf

Plus, on macOS:

git config --global core.autocrlf input

5. Fixing an already-mixed repo

Do this as its own commit, with nothing else in it:

git add --renormalize .
git commit -m "Normalise line endings"

Why it must be isolated: mixed with content changes, a normalisation commit becomes unreviewable and hides the real edits inside a sea of line-ending churn.

6. Telling a line-ending diff from a real one

If the first shows the whole file changed and the second shows nothing, it is purely line endings — no real content moved. If the second still shows lines, that part is a genuine change:

git diff --stat
git diff --ignore-cr-at-eol

7. Practical warning

Check line endings on BOTH branches before moving files between them. If one side is CRLF and the other LF, every line of every moved file reads as changed — and the move (section above) then looks like a full rewrite to Sonar.
